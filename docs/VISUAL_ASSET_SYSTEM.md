# Visual Asset System — ITTYBITTYBITES Library

_Build Order:_ Library Season 1 — Asset Pipeline  
_Date:_ 2026-07-08  
_Status:_ Asset system established & registered

> This document is the canonical reference for **where visual assets live, how they are named, and how future agents should add or integrate them.** It supersedes nothing in `docs/ILLUSTRATION_SYSTEM.md` — it extends that system with a production-scale asset layer.

---

## 1. Purpose & scope

This pass establishes the **production visual asset system** for Library Season 1:

- a 3840×1600 **library hero**,
- **collection covers** (2400×1600) + **thumbnails** (600×400) for all 8 collections,
- **experience heroes** (1600×900) + **thumbnails** (1200×900) for the 8 featured experiences,
- a **per-asset registry** (`ASSET_MANIFEST.json`) so no asset is ever created twice,
- this documentation, including exact integration points for future wiring.

It deliberately **extends** the existing illustration system rather than replacing it. No foundation files, experience logic, scoring, routing, or content models were modified.

## 2. Two asset roots (read this first)

The repository has **two** `public/assets/` roots. They are intentional and complementary:

| Root | Role | Generated by | Referenced by live code? |
|------|------|--------------|--------------------------|
| `public/assets/library-season-1/` | **Base set** — icons, badges, patterns, backgrounds, and the small editorial illustrations used across cards and detail pages. | `scripts/generate-library-season-1-assets.mjs` | **Yes** — `src/platform/collection-identity.ts` and `src/platform/illustration-system.ts` resolve these paths at runtime. Do **not** rename or move these files. |
| `public/assets/library/` | **Production set** — hero, collection covers, and experience heroes/thumbnails at full resolution. | `scripts/generate-library-production-assets.mjs` | Not yet wired into components (see §8 — integration is prepared, not applied). |

**Why two roots?** The base set was already in production and is hard-referenced by live code under the flat `library-season-1/` layout. The brief required a new canonical tree (`assets/library/{category}/{id}/…`) for production-scale art. Creating the new art in the mandated tree — without disturbing the live base set — avoids broken references and avoids duplicate systems (the production generator **imports** the base generator's motif functions, so there is one source of visual truth).

```
public/assets/
├── library-season-1/            # BASE SET (live, do not move)
│   ├── icons/                    #   {collection}.svg, bookmark.svg
│   ├── collection-badges/        #   {collection}.svg
│   ├── patterns/                 #   {collection}.svg
│   ├── backgrounds/              #   {collection}.svg
│   ├── illustrations/
│   │   ├── collections/          #   {collection}.svg  (640x420)
│   │   └── experiences/          #   {experience}.svg  (320x220)
│   └── asset-index.json          #   folder-level index
│
└── library/                      # PRODUCTION SET (this pass)
    ├── hero/
    │   └── library-season-1/
    │       └── library-hero.svg          # 3840x1600 (12:5)
    ├── collections/
    │   └── {collection}/
    │       ├── cover.svg                 # 2400x1600 (3:2)
    │       └── thumbnail.svg             # 600x400  (3:2)
    ├── experiences/
    │   └── {experience-id}/
    │       ├── hero.svg                  # 1600x900 (16:9)
    │       └── thumbnail.svg             # 1200x900 (4:3)
    └── ASSET_MANIFEST.json               # per-asset registry (ALL assets)
```

**For future seasons:** add new production art under `public/assets/library/{category}/{season-or-id}/…` and register it in `ASSET_MANIFEST.json`.

## 3. Naming convention

**Logical name:** `category-purpose-variant.extension`

Examples:

```
collection-foundations-cover.svg
experience-e003-optimization-thumbnail.svg
pattern-mathematics-symmetry.svg
icon-discovery.svg
```

**Physical placement:** production assets realize the logical name as a path, where the directory already encodes category + id and the file carries the purpose:

```
assets/library/collections/foundations/cover.svg
assets/library/experiences/e003-optimization/thumbnail.svg
```

This avoids long, repetitive filenames while keeping the `category-purpose-variant` intent recoverable from the path. Legacy base assets keep their flat `{id}.svg` names (they predate this convention and are referenced by live code).

**Experience IDs:** the 8 featured experiences use the brief's prefixed editorial IDs, which encode collection + sequence. The canonical platform slug is preserved in the manifest's `usage` field for cross-reference:

| Editorial ID | Canonical slug | Collection |

| `h001-dueling-accounts` | `dueling-accounts` | history |
| `c002-constraint` | `constraint` | creativity |
| `e001-bridge-builder` | `bridge-builder` | engineering |
| `e003-optimization` | `optimization` | engineering |
| `x001-attention` | `attention` | society-mind |
| `m005-proof` | `proof` | mathematics |
| `n005-watershed` | `watershed` | nature |
| `f003-pattern-garden` | `pattern-garden` | foundations |

## 4. Dimensions & formats

| Asset | Dimensions | Aspect | Format |
|---|---|---|---|
| Library hero | 3840 × 1600 | 12:5 | svg |
| Collection cover (master) | 2400 × 1600 | 3:2 | svg |
| Collection thumbnail | 600 × 400 | 3:2 | svg |
| Experience hero | 1600 × 900 | 16:9 | svg |
| Experience thumbnail | 1200 × 900 | 4:3 | svg |

**SVG requirements (enforced):**

- optimized paths only — no embedded raster images, no `base64`, no `<image>`;
- **no `<text>` elements** (labels are HTML; the generator strips any inherited `<text>` so production SVGs stay clean);
- accessible `<title>`/`<desc>` on every asset;
- scalable `viewBox`;
- lightweight (largest file ≈ 11 KB; entire production set ≈ 60 KB of SVG).

## 5. The asset registry — `public/assets/library/ASSET_MANIFEST.json`

**Every asset in the system is registered here** — both the new production set and the existing base set. Query this file **before** creating any asset.

Each entry has exactly seven fields (matching the brief's schema):

```json
{
  "asset": "public/assets/library/collections/foundations/cover.svg",
  "type": "cover",
  "collection": "foundations",
  "purpose": "collection-cover",
  "dimensions": "2400x1600",
  "format": "svg",
  "usage": "Master cover artwork for the Foundations collection. …"
}
```

The file also carries envelope metadata: `schemaVersion`, `generatedAt`, `season`, `namingConvention`, `roots`, `counts`, and `fields`.

**Types in use:** `hero`, `cover`, `thumbnail`, `icon`, `badge`, `pattern`, `background`, `illustration`, `index`, `manifest`.

The manifest is regenerated by the production generator (it walks `library-season-1/` and merges with the production assets it writes), so it always reflects the real on-disk state.

## 6. How to add an asset (future agents)

1. **Check the registry first.** Open `public/assets/library/ASSET_MANIFEST.json` and search for the collection + purpose you need. If a suitable asset exists, reuse it.
2. **Pick the right root.** Icons/badges/patterns/backgrounds/small illustrations belong in the **base** set (`library-season-1/`, generated). New full-resolution hero/cover/experience art belongs in the **production** set (`library/`).
3. **Follow the naming + placement rules** in §3.
4. **Prefer extending the generator** over hand-authoring one-off files, so art stays cohesive with the motif system.
5. **Regenerate + re-register.**
   - Base motifs: edit `src/content/assets/illustration-manifest.json` (palette/motif/caption is the single source of truth), then `node scripts/generate-library-season-1-assets.mjs`.
   - Production assets: `node scripts/generate-library-production-assets.mjs` (re-imports the motifs and rewrites `ASSET_MANIFEST.json`).
6. **Validate:** `npm run lint && npm run build && npm test`.

|---|---|---|
| foundations | Geometry, grids, structure, precision | grid-frame |
| history | Maps, paper, archives, timelines | archive-sheet |
| science | Observation, instruments, measured uncertainty | instrument-grid |
| nature | Ecosystems, flows, branching living cycles | branch-current |
| creativity | Layered composition, sketch, expressive assembly | collage-studio |
| engineering | Blueprint language, load paths, precision | blueprint-span |
| mathematics | Elegant abstraction, symmetry, proof, curves | curve-proof |
| society-mind | Conversation, networks, shifting perspective | network-dialogue |

Design principles (from `docs/ILLUSTRATION_SYSTEM.md`): **explanatory not decorative; editorial not app-like; reusable across seasons; lightweight by default.**

## 8. Integration points (prepared, not yet wired)

The production assets are **created and registered** but **not yet rendered** by components — this pass was scoped to the asset pipeline only. Future integration should happen at the existing render seam, which keeps foundation/experience/scoring/routing untouched.

### The render seam (platform helpers)

| File | Existing helper | Renders |
|---|---|---|
| `src/platform/collection-identity.ts` | `renderCollectionIllustration(collectionId, w, h, cls)` | base collection illustration |
| `src/platform/collection-identity.ts` | `renderCollectionBadge(collectionId, size, cls)` | base collection badge |
| `src/platform/illustration-system.ts` | `renderExperienceArtwork(expId, {…})` | base experience illustration |
| `src/platform/illustration-system.ts` | `renderExperienceFigure(expId, title)` | experience hero figure w/ caption |

### Recommended integration approach

Add **new, opt-in** helpers alongside the existing ones (do not change existing signatures) that resolve production paths, e.g.:

- `renderCollectionCover(collectionId, {variant:'cover'|'thumbnail'})` → `assets/library/collections/{id}/{variant}.svg`
- `renderExperienceHero(expId, {variant:'hero'|'thumbnail'})` → `assets/library/experiences/{id}/{variant}.svg`
- `renderLibraryHero()` → `assets/library/hero/library-season-1/library-hero.svg`

…resolving the editorial→canonical experience id via the table in §3, and reading dimensions from `ASSET_MANIFEST.json`.

### Exact call sites to upgrade (when wiring)
